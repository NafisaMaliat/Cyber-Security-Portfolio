Lab 5 — Web Application Security: WPScan

Learning Outcomes: LO1, LO3  |  WordPress Vulnerability Assessment
Attacker VMKali Linux (ARM64) — 192.168.123.100
Target VMWordPress Ubuntu Server — 192.168.123.30
NetworkUTM Shared Network (192.168.123.0/24)
ToolWPScan

Methodology

Step 1

Network Verification

Verified connectivity between Kali and WordPress target using ICMP ping tests. Confirmed 0% packet loss on the 192.168.123.0/24 subnet.

Step 2

Database Update & Enumeration

Updated WPScan's vulnerability database, then enumerated WordPress components including themes and plugins.

wpscan --url http://192.168.123.30 --enumerate vp,vt
Step 3

User Enumeration

Discovered valid usernames "student" and "nafisa" via WordPress REST API and author archive scanning.

wpscan --url http://192.168.123.30 --enumerate u
Step 4

Brute Force Attack

Dictionary-based password attack against the nafisa account. Successfully cracked password "4444" — a trivially weak credential.

wpscan --url http://192.168.123.30 --passwords passwords.txt --username nafisa
Step 5

API Vulnerability Scan & Throttled Scan

Used WPScan API for CVE cross-referencing. Demonstrated throttled scanning for production-safe assessment.

Key Findings

Weak Password

Password "4444" cracked instantly via dictionary attack. Full account compromise achieved.

User Enumeration

WordPress exposed valid usernames through REST API and login error messages.

Plugin Exposure

Theme and component versions revealed, enabling targeted CVE exploitation.

No Rate Limiting

No CAPTCHA or brute-force protection observed on login endpoint.

Lab 8 — Vulnerability Assessment: Nmap & SSH

Learning Outcomes: LO1, LO2  |  Network Scanning, Service Detection & SSH
AttackerKali Linux (ARM64) — 192.168.123.100
GatewayUbuntu Gateway — 192.168.123.1
TargetWordPress Ubuntu Server — 192.168.123.30
ToolsNmap, SSH, SCP, SFTP, Nikto, curl

Methodology

Phase 1

Host Discovery & Port Scanning

Ping sweep confirmed 3 live hosts. Basic scan found ports 22 (SSH) and 80 (HTTP) open on the WordPress target.

nmap -sn 192.168.123.0/24
Phase 2

Service & Version Detection

Identified OpenSSH 8.9p1 on port 22 and Apache 2.4.52 on port 80. Aggressive scan revealed WordPress 6.9.4.

nmap -A 192.168.123.30
Phase 3

SSH Validation & Tunnelling

Successful login, RSA 4096-bit key deployment, SCP/SFTP file transfers, and local/remote port forwarding demonstrated.

ssh -L 80:localhost:80 student@192.168.123.30
Phase 4

Web Assessment with Nikto

Application-layer assessment revealed missing security headers, browsable uploads directory, and WordPress information disclosure.

nikto -h http://192.168.123.30

Key Findings

OpenSSH 8.9p1

Version exposed on port 22 — enables targeted CVE lookup.

Apache 2.4.52

Web server version visible in headers and scan output.

WordPress 6.9.4

CMS identity and version confirmed via aggressive scan.

Missing Headers

X-Frame-Options and X-Content-Type-Options absent.

Possible CSRF

Nmap vuln script flagged WordPress form without token.

IPv6 Exposure

SSH and HTTP accessible over both IPv4 and IPv6.

Lab 9 — Vulnerability Scanning & Mitigation: Nessus

Learning Outcomes: LO1, LO2  |  Nessus on ARM64, CVE Triage & Compensating Controls
Scanner HostKali Linux (ARM64) — 192.168.123.100
GatewayUbuntu Gateway — 192.168.123.1
Target 1Ubuntu Desktop — 192.168.123.40
Target 2WordPress Ubuntu Server — 192.168.123.30
Nessus VersionEssentials Plus v10.11.3 (aarch64)

Scan Results Summary

Kali — ImageMagick (CVSS 7.8)

Multiple vulnerabilities in ImageMagick < 7.1.2-16. Version-based detection triggered.

Kali — ImageMagick (CVSS 7.5)

Additional issues requiring upgrade to 7.1.2-16 or later.

CVE-2025-62171 (CVSS 5.5)

Integer overflow affecting 32-bit builds only. ARM64 system not vulnerable — analyst validation required.

Ubuntu Desktop

Auth: Fail, 1 Low, 4 Informational — limited uncredentialed findings.

WordPress Server

Auth: Fail, 1 Low, 27 Informational — web service enumeration data.

Mitigation Workflow

Step 1

Attempted Package Update

apt reported ImageMagick already at newest version. Patch gap confirmed — fixed version not yet in repository.

sudo apt update && sudo apt install imagemagick
Step 2

Compensating Control — policy.xml

Restricted vulnerable coders (BMP, PNG, TIFF) via ImageMagick security policy to block risky processing paths.

<policy domain="coder" rights="none" pattern="BMP" />
Step 3

Verification

BMP conversion attempt returned "not allowed by the security policy" — confirming the compensating control is active.

convert logo: test.bmp → BLOCKED
