Lab 5 — Web Application Security: WPScan

Learning Outcomes: LO1, LO3 | WordPress Vulnerability Assessment

1. Introduction

This lab focuses on testing the security of a WordPress installation using WPScan, a vulnerability scanner designed specifically for WordPress. The goal was to perform reconnaissance on the target, identify WordPress users, check installed components, and try password attacks using a dictionary list. This exercise shows how automated tools can find vulnerabilities in poorly secured web applications, and it highlights why strong passwords and secure configurations are so important.

2. Lab Environment

ComponentDetails
Host OSmacOS (Apple Silicon / ARM64)
Attacker VMKali Linux (ARM64) — 192.168.123.100
Target VMWordPress Ubuntu Server — 192.168.123.30
NetworkUTM Shared Network (192.168.123.0/24)
Security ToolWPScan

The target was hosted on a private virtual network to make sure the testing was safe, ethical, and authorised. Network Architecture: Internet → Gateway VM → Kali Linux (Attacker: 192.168.123.100) → WordPress Server (Target: 192.168.123.30)

3. Methodology

Verified connectivity between Kali and WordPress target using ICMP ping tests. Confirmed 0% packet loss on the 192.168.123.0/24 subnet.

Enumerated WordPress components including themes and plugins to identify installed packages and versions.

5. Recommended Security Improvements

  1. Enforce strong password policies with minimum 12+ characters, uppercase, lowercase, numbers, and symbols
  2. Implement rate limiting and CAPTCHA on login endpoints to prevent brute force attacks
  3. Disable user enumeration via REST API (/wp-json/wp/v2/users endpoint)
  4. Hide WordPress version and component information from HTTP headers and source code
  5. Keep WordPress, themes, and plugins updated to the latest patch versions
  6. Implement Web Application Firewall (WAF) to detect and block automated scanning tools
  7. Enable two-factor authentication for all administrator accounts
  8. Monitor and log all authentication attempts and unusual activities

6. Reflection

This lab demonstrated the power of automated vulnerability scanning tools and how quickly even a sophisticated application like WordPress can be compromised if basic security practices are not followed. The combination of weak passwords, exposed user information, and lack of brute-force protection created a perfect storm for account takeover.

The three-step attack pattern was straightforward: (1) Enumerate users, (2) Crack weak password, (3) Gain admin access. In a real-world scenario, the attacker could then install backdoors, deface the website, steal data, or use the server as a launching point for further attacks.

I learned that security is not just about deploying tools — it's about implementing defence in depth. Even a single strong password with rate limiting would have completely blocked this attack. This lab reinforced the principle that attackers only need one weakness to succeed, while defenders must protect against all potential attack vectors.

The use of automated tools like WPScan should always be done responsibly within authorised scope. These same tools are essential for legitimate security testing, but require proper access controls, clear documentation, and adherence to ethical guidelines.

7. Ethical Considerations

All testing was conducted within an isolated, authorised lab environment. The WordPress server was a learning target specifically set up for security training. No real systems or data were targeted. The methodology followed responsible disclosure principles and OWASP guidelines for ethical hacking.

Lab 7 — Web Application Vulnerabilities: OWASP Juice Shop

Learning Outcomes: LO1, LO2, LO3 | 11 Challenges Completed

Aim

To identify and exploit web application vulnerabilities in the OWASP Juice Shop application, understanding real-world attack vectors and mitigation strategies.

Introduction

OWASP Juice Shop is a deliberately vulnerable web application designed for security training. It teaches common vulnerabilities including cross-site scripting (XSS), broken access control, security misconfiguration, and sensitive data exposure. The application mimics a real e-commerce platform while containing intentional flaws for learning purposes.

This lab completed 11 out of 173 available challenges, covering a variety of vulnerability categories and attack techniques. Each challenge required manual inspection, code review, and exploitation of specific security weaknesses.

Observations

The application appeared normal at first glance but contained numerous intentional security flaws. DevTools proved invaluable for discovering hidden endpoints, monitoring API calls, and identifying exposed sensitive information. Many challenges could not be solved by using the application normally—they required modifying requests, accessing hidden pages, or manipulating client-side validation. Several challenges involved chained exploits, reflecting real-world complexity.

Discussion

Multiple vulnerabilities coexisting in a single application stemmed from basic security oversights: missing validation, unprotected endpoints, verbose error messages, and client-side security controls. Reconnaissance (inspecting storage, source code) often revealed enough information to compromise the application, aligning with the principle that attackers take the path of least resistance.

The Zero Stars challenge was the clearest example of "Never trust client-side validation." Browser prevention failed to stop simple HTTP request bypassing. This principle applies to all client-side security: authentication checks, authorisation decisions, rate limiting, and input validation must always be server-side.

The experience highlighted why security training is essential for developers. Many vulnerabilities could be completely prevented with security-aware development practices, code review processes, and proper testing before deployment.

Conclusion

Completing 11 challenges reinforced core web security principles. Each challenge told a story about a different security failure. Rather than collecting vulnerability names, understanding why each vulnerability was dangerous and how attackers exploit it in practice provided real value.

I would recommend Juice Shop to any developer learning security—it provides a safe environment to experiment with attacks, understand vulnerabilities deeply, and build intuition about how real systems fail.

Reflection

This lab was enjoyable because each challenge was engaging and educational. The gamified scoreboard made the experience fun while maintaining educational value. One surprising insight: unminified JavaScript reveals system architecture, APIs, and sometimes credentials. Developers often assume JavaScript is hidden because it's client-side, but it's fully visible and analysable.

The lab reinforced that vulnerability scanning is a starting point—humans must verify, analyse impact, and decide remediation. Automation helps identify issues; human insight determines criticality.

Lab 8 — Vulnerability Assessment: Nmap & SSH

Learning Outcomes: LO1, LO2 | Network Scanning, Service Detection & SSH

Abstract

Comprehensive network reconnaissance and vulnerability assessment on a WordPress server using Nmap for enumeration, SSH for access and tunnelling, and Nikto for web assessment. Multiple scanning techniques tested including ping sweeps, port scans, OS fingerprinting, and service version detection. SSH tunnelling and secure file transfer demonstrated practical exploitation techniques. Findings revealed multiple attack surfaces and misconfigurations.

Introduction

Network reconnaissance is the first phase of security assessment. This lab combined complementary tools: Nmap for network-level reconnaissance, SSH for system-level access and tunnelling, and Nikto for application-level assessment.

Aim and Objectives

Perform complete vulnerability assessment using network scanning, service enumeration, and web assessment techniques. Objectives include: discovering active hosts, identifying open ports, determining service versions, performing SSH validation and tunnelling, conducting web application assessment, and identifying security misconfigurations.

Discussion

The assessment revealed a typical small-to-medium business setup lacking security headers and running unpatched versions. However, basic practices (no defaults, SSH support) prevented trivial attacks. Nmap and Nikto provided complementary views—network vs. application level. Thorough assessment requires multiple tools and analysis at different stack layers.

SSH access demonstrated system-level compromise potential and how tunnelling hides activities from network monitoring. Multiple findings were exploitable in real attacks: missing headers enable XSS, weak CSRF enables social engineering, lack of detection leaves systems blind to reconnaissance.

Conclusion

Lab successfully demonstrated complete reconnaissance and vulnerability assessment workflow from initial discovery to detailed analysis. Methodology revealed multiple security weaknesses enabling compromise. Lab emphasized that attackers and defenders use the same tools—authorization and intent differ.

Reflection

Lab was valuable combining tools with practical exploitation. Hands-on experience interpreting Nmap results beats documentation reading. SSH tunnelling concept of encrypting traffic through compromised systems was intellectually interesting and practically powerful. Recognized where different tools fit: Nmap excels at rapid enumeration; Nikto better for detailed application assessment. Combining multiple tools requires understanding strengths/limitations.

Lab reinforced that assessment is craft not checklist. Running tools is easy; interpreting results and determining impact requires experience.

Analysis

Lab identified realistic vulnerability management scenario: critical vulnerabilities exist, patches unavailable. Many organisations face this situation. Rather than accepting risk, compensating controls reduce exploitability. policy.xml mitigation exemplified defence-in-depth: even with ImageMagick bug, restricting vulnerable coders prevents exploitation. Not perfect—determined attackers might find alternatives—but substantially reduces risk until proper patches available.

Multi-target scanning demonstrated comprehensive assessment across environment. Vulnerability management requires understanding complete attack chain, not just collecting CVE numbers.

Conclusion

Lab successfully demonstrated enterprise vulnerability management: deployment of professional scanning tools, interpretation of detailed reports, prioritisation of findings, and implementation of remediation workflows. Emphasis on compensating controls when patching impossible showed that perfect solutions rare—security professionals work with available resources to reduce risk.

Reflection

Deploying Nessus on ARM64 straightforward—modern security tools support diverse architectures. Network troubleshooting reminded me scanning reveals infrastructure issues beyond vulnerabilities. Appreciated emphasis on compensating controls. Vulnerability management not binary—when patching impossible, alternative mitigations reduce risk pragmatically. CVSS scores useful for triage but insufficient alone—reading CVE details essential. Automation helps; human analysis critical.

Lab reinforced vulnerability scanning starting point not complete assessment. Humans verify, analyse impact, decide remediation.
